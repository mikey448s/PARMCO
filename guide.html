1.0 Hardware Topography & Assembly

The physical layer of the PARMCO system requires strict adherence to logic-level routing. The Raspberry Pi 4B interacts with high-current 12V DC systems; proper optical isolation is mandatory.

FAILURE CASE: PiHat 1.8V Logic Starvation

Observation: The breakout board stepped all RPi4B GPIO pins down to 1.7V. This was insufficient to drive the 4N25 optocoupler's internal LED (forward voltage ~1.2V, but required ~10mA of current), resulting in a failure to trigger the MOSFET.

RESOLUTION: 2N3904 Current Sink

Implementation: The 4N25 anode was re-routed directly to the Pi's continuous 3.3V power rail via a 100Ω resistor. A 2N3904 NPN transistor was placed between the 4N25 cathode and ground. GPIO 17 now drives the base of the NPN. The 1.7V logic easily overcomes the NPN's 0.7V Vbe threshold, snapping the switch closed and sinking the required current.

FAILURE CASE: MOSFET Gate Sub-Saturation

Observation: The motor failed to spin. Amperage draw at the power supply measured 5mA. The initial 1KΩ pull-down resistor on the IRFZ34N gate was drawing too much current, causing the 4N25 output to collapse to ~3V (below the IRFZ34N enhancement threshold).

RESOLUTION: Impedance Adjustment

Implementation: Replaced the 1KΩ gate pull-down with a 100KΩ resistor. Gate voltage successfully rose to ~11.5V, achieving full MOSFET saturation.

2.0 RPi4B C-Core Execution Environment

The system daemon relies on low-level hardware memory access. Standard user privileges are insufficient.

CRITICAL OPTIMIZATION: Glitch Filter Removal

Observation: At speeds exceeding 10% duty cycle, the `pigpio` hardware glitch filter (gpioGlitchFilter) classified the rapid pulses of the physical fan blades as electromagnetic noise, skipping critical RPM data.

Implementation: The hardware filter was disabled. A software-based state-tracking debounce was implemented within the `ir_sensor_callback` ensuring microsecond resolution without bouncing: if (tick - last_tick > 2000).

CRITICAL OPTIMIZATION: Stiction Floor & Damping

Observation: Static friction required >10% duty cycle to break, rocketing the motor to 1800 RPM. The PID loop panicked and cut power, causing an endless stall-to-rocket feedback loop at low RPMs.

Implementation: Implemented a 6.2% hard stall floor to prevent the motor from ever entirely losing kinetic momentum during an overcorrection, and added a 5x loop dampener when operating below 12.5% duty cycle.

Compilation Directives

Ensure the `pigpio` and `libbluetooth-dev` dependencies are installed via apt. Compile the core using the following flags to link the necessary math, realtime, and bluetooth libraries:

gcc -o parmco_server main.c -lpigpio -lbluetooth -lrt -pthread -lm

To execute, the `pigpiod` daemon must be running, and the server must be launched as root:

sudo pigpiod
sudo ./parmco_server

3.0 Android Mobile Command Client

The remote interface is developed in Kotlin utilizing Jetpack Compose for declarative UI rendering and Coroutines for background socket telemetry listening.

FAILURE CASE: Bluetooth Service Discovery Rejection

Observation: The standard Android implementation utilizing device.createRfcommSocketToServiceRecord(UUID) resulted in connection timeouts. The Linux BlueZ stack on the RPi4B rejected the SPP inquiry.

RESOLUTION: Reflection-Based Port Hijacking

Implementation: The standard API was bypassed utilizing Java reflection. The application forces an insecure RFCOMM connection directly onto Port 1, matching the hardcoded port of the C-server.

val method = device.javaClass.getMethod("createRfcommSocket", Int::class.javaPrimitiveType)
val socket = method.invoke(device, 1) as BluetoothSocket
