LOG 007 // MATH: KINEMATIC TRANSFER FUNCTION
2026-04-20
RESOLVED [EMPIRICAL FIT]
SYMPTOM
The legacy 5th-order polynomial transfer function fed massive inaccuracies to the PID loop. A 9.5% commanded duty cycle (expected 1000 RPM) physically yielded ~2000 RPM on the new motor setup.
ANALYSIS
Hardware differences invalidated the previous semester's characterization curve. The C-core Feed-Forward math was entirely misaligned with physical reality.
DECISION
Conducted empirical Hardware-in-the-Loop (HIL) testing. Truncated the complex 5th-order polynomial down to a highly accurate Quadratic Equation: Duty = (0.00000125 * RPM²) - (0.0011 * RPM) + 6.7.
LOG 006 // PHYSICS: KINETIC STICTION BATTLE
2026-04-18
RESOLVED [SERVER V1.5]
SYMPTOM
At low RPM targets, the motor refused to spin until the loop aggressively ramped up power. Once stiction broke, the motor rocketed to 1800 RPM. The PID panicked, dropped power below the kinetic floor, and the motor stalled. Cycle repeated infinitely.
ANALYSIS
The physical difference between Static Friction (getting started) and Kinetic Friction (staying spinning) created a dead-zone that the linear PID loop could not resolve without stalling.
DECISION
Enforced a hard 6.2% minimum duty cycle "Stall Floor". Additionally, decoupled the PID step size: when operating below 12.5% duty cycle, the error multiplier is artificially reduced by 5x to prevent violent overcorrections.
LOG 005 // SOFTWARE: RPM SIGNAL DELETION
2026-04-10
RESOLVED [SERVER V1.2]
SYMPTOM
RPM feedback functional at low motor speeds, but abruptly failed as speed increased >10% duty cycle. Sensor read 0 RPM despite physical blade rotation.
ANALYSIS
The C program utilized an aggressive pigpio hardware glitch filter set to 1ms. At higher speeds, fan blades passed the sensor faster than 1ms. The Pi correctly classified valid blade-passes as high-frequency noise and deleted the data.
DECISION
Disabled the hardware filter entirely. Implemented a sophisticated C-Core State Debounce directly in the interrupt callback function: if (tick - last_tick > 2000).
LOG 004 // SOFTWARE: ANDROID BLUETOOTH REJECTION
2026-04-01
RESOLVED [CLIENT V1.1]
SYMPTOM
Android App failed to establish an RFCOMM socket connection with the RPi4B. The connection timed out on UUID service discovery.
ANALYSIS
The Linux BlueZ stack on the RPi4B was failing to correctly advertise the Serial Port Profile (SPP) UUID to the Android SDP inquiry.

PROTOCOL HIJACK. Kotlin code was rewritten to utilize Java Reflection, forcefully instantiating an insecure socket directly onto hardcoded RFCOMM Port 1.
⚠️ WARNING 004: This bypass may fail on newer Android OS versions with stricter reflection restrictions or against BlueZ stacks running customized security policies.
LOG 003 // HARDWARE: MOSFET GATE UNDER-SATURATION
2026-03-22
RESOLVED [REV G]
SYMPTOM
Keysight bench power supply indicated only a 5mA draw during the 'enable' phase. Motor failed to spin under load.
ANALYSIS
The 5mA draw indicated the 12V rail was not reaching the IRFZ34N gate. The initial 1KΩ pull-down resistor was drawing excessive current, collapsing the 4N25 phototransistor output to ~3V.
DECISION
CRITICAL IMPEDANCE CHANGE. Replaced the 1KΩ gate pull-down with a 100KΩ resistor. Phototransistor output successfully reached ~11.5V, achieving full MOSFET saturation.
LOG 002 // HARDWARE: LOGIC LEVEL STARVATION
2026-03-15
RESOLVED [REV G]
SYMPTOM
Motor failed to actuate when GPIO 17 was commanded HIGH. Probing revealed the PiHat reduced GPIO logic levels from 3.3V to ~1.7V.
ANALYSIS
A 1.7V signal is insufficient to consistently drive the LED forward voltage (Vf) inside the 4N25 optocoupler while meeting current requirements (~10mA).
DECISION
Implemented a 2N3904 NPN transistor as a current-sinking switch. The weak 1.7V GPIO signal easily saturates the NPN, closing the current path for the 4N25 LED.
LOG 001 // CONCEPTUAL: DRIVER TOPOLOGY PIVOT
2026-03-01
SUPERSEDED
PROBLEM
Initial conceptual schematic utilized an L298N H-Bridge module. AI review flagged a violation: the L298N uses internal Bipolar Darlington transistors.
CONSTRAINT
Project Specifications explicitly require a MOSFET driver topology for efficiency.
DECISION
Immediate pivot from L298N integrated circuit to a discrete H-Bridge design utilizing N-Channel Power MOSFETs (IRFZ34N).
